# `read_logs(editor)` 修复留档

本文记录 `read_logs` 工具在 `editor` 分支上的两轮关键修复：

1. 传输层噪声混入编辑器日志，导致返回内容不稳定、不利于定位真正问题。
2. 富文本/ANSI 控制序列污染日志内容，在大批量回读时触发解析与传输超时。

目标不是重复实现细节，而是保留这次修复中已经验证过的约束、判断条件和验证方法，供后续新增或调试自定义工具时复用。

## 适用范围

适用于这条链路上的所有“跨 TypeScript → WebSocket → Godot 编辑器”的自定义工具，尤其是：

- 返回大量文本或数组数据的工具
- 读取编辑器内部状态、面板输出、缓存或日志的工具
- 有多个分支、多个数据源或多个过滤条件的工具
- 任何出现“部分可用、部分超时”症状的工具

## 现象

`read_logs` 的 `runtime` 分支可以正常返回，但 `editor` 分支先后出现过两类问题：

### 1. 返回内容夹杂传输噪声

编辑器日志里混入了插件自身的通信输出，例如：

- `[Client ...] ...`
- `Sending response to client ...`
- `Processing command: ...`
- `Executing script... ready func`
- `result_data: ...`

结果：

- MCP 返回中出现与用户目标无关的噪声
- 真正的错误、警告和脚本输出被淹没
- 测试结果不稳定，难以判断工具本身是否正常

### 2. 大批量回读超时

当日志包含富文本颜色标记、ANSI 序列或控制字符时，`read_logs(editor)` 在较大 `count` 下会出现超时或输出异常。

结果：

- 单条或少量读取可能正常
- 增大 `count` 后开始超时
- 同一条日志在编辑器面板可见，但经 MCP 回读后内容被污染

## 根因

这次修复里确认了两个直接根因。

### 根因 A：采集源没有区分“业务日志”和“传输日志”

编辑器侧 logger 会同时接收到：

- 用户真正关心的 Warning / Error / Script / General
- MCP 插件自身为调试连接和命令处理打印的输出

如果不在采集阶段过滤，后续分页、类型筛选和摘要格式化都无法补救。

### 根因 B：日志内容未经标准化就进入缓存与返回链路

富文本颜色、ANSI 转义序列和不可见控制字符会污染：

- Godot 侧内存缓存
- JSON 序列化后的消息体
- TypeScript 侧摘要拼接

问题不一定在“小样本”下立即暴露，但在大批量读取、连续读取或混合类型日志中会迅速放大。

## 已落地的修复策略

### 1. 在编辑器 logger 采集阶段做标准化

位置：`addons/godot_mcp/mcp_server.gd`

处理顺序：

1. 去除 ANSI 序列
2. 去除控制字符
3. `strip_edges()` 清理首尾空白
4. 空消息直接丢弃

原因：

- 问题源头在 Godot 侧采集，越早清洗，后续缓存、分页和返回越稳定
- 不把污染内容写进 `_entries`，才能避免后续每次读取都重复承担清洗风险

### 2. 在采集阶段过滤传输噪声

位置：`addons/godot_mcp/mcp_server.gd`

当前过滤条件基于前缀判断，已验证需要过滤的噪声包括：

- `[Client `
- `Sending response to client `
- `Processing command: `
- `Executing script... ready func`
- `result_data: `

原则：

- 过滤“插件自身的运输/调试输出”
- 保留真正有诊断价值的用户脚本错误、警告和通用输出
- 过滤条件应基于稳定前缀，而不是模糊包含，避免误伤正常日志

### 3. 维持固定容量的内存缓存

位置：`addons/godot_mcp/mcp_server.gd`

- 使用 `_entries` 保存结构化条目
- 用 `MAX_ENTRIES` 控制上限
- 超出上限时移除最旧条目
- 条目带连续递增 `index`

原因：

- 避免 editor 日志无限增长后拖慢回读
- 让分页和排序以“已采集缓存”为准，而不是每次重扫编辑器面板
- `index` 可作为调用侧验证顺序和缺失的稳定依据

### 4. 在 TypeScript 侧做兜底清洗

位置：`server/src/tools/debug_tools.ts`

虽然主要清洗已经前置到 Godot logger，但 TypeScript 侧仍保留 `sanitizeLogText()` 兜底，继续去除：

- ANSI 序列
- 残留颜色片段
- 控制字符

原则：

- 主要修复放在源头
- 展示层仍保留轻量兜底，防止历史条目或异常输入再次污染返回结果

## 这次修复提炼出的通用条件

后续做自定义工具时，满足下列任一条件，就不要直接“读完再返回”，而应优先设计缓冲、过滤和标准化：

- 返回内容可能超过几十条文本记录
- 数据源包含编辑器内部日志、终端输出、脚本打印或第三方返回文本
- 同一工具有多个 `source` / `mode` / `branch`
- 输出中可能包含富文本、颜色标记、二进制片段或不可见字符
- 结果需要做分页、排序、类型筛选或增量回读

## 推荐的错误处理方式

### 参数层

- 用 TypeScript Zod 在入口限制 `source`、`type`、`count`、`offset`、`order`
- 参数概念上必填，就不要依赖 Godot 侧兜底默认值
- `count` 上限必须明确限制，避免一次拉取过大结果

### 分支层

- 每个分支都必须独立返回 `_send_success()` 或 `_send_error()`
- 不允许“某个分支可用就视为工具可用”
- 空结果可以是成功，但必须显式返回 `logs: []` 和 `total_count`

### 数据层

- 文件不存在、插件未挂载、接口不可用时，返回明确错误
- 类型不匹配、offset 越界、过滤后为空，不应报错，应返回空集合
- 文本进入缓存前先标准化，不把脏数据留给展示层处理

### 超时层

如果某个工具出现超时，优先按下面顺序排查：

1. 是否所有路径都能返回 `_send_success()` / `_send_error()`
2. 是否有某一支分支单独失败，而非整个工具都失败
3. 是否在处理大 payload、富文本或污染字符时放大问题
4. 是否把调试输出再次写回了当前被读取的数据源
5. 是否缺少分页、容量上限或最小化返回策略

## 这次修复后的验证口径

这轮修复最终按下面的口径验证：

1. 重新注入当前运行实例的 editor logger 实现
2. 单独验证 `read_logs(editor)` 的噪声过滤效果
3. 单独验证 `read_logs(runtime)` 没有被回归破坏
4. 生成高数量 editor 日志，验证较大 `count` 下仍能正常回读
5. 验证 rich 文本与控制字符场景不再触发超时

只有同时满足以上几点，才认为 `read_logs` 真正完成，而不是“有一个分支能跑通”。

## 对后续工具实现的直接约束

- 命令名一致性仍然是第一前提，但不是完成条件
- 任何多分支工具都要逐分支实测
- 任何返回文本集合的工具都要先考虑标准化与限流
- 采集层发现的噪声应在采集层解决，不要把问题推迟到展示层
- 调试打印不要与业务返回共用同一数据源，必要时单独隔离
